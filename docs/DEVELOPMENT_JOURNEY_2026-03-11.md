# xstrm-suite 项目折腾过程记录（2026-03-11）

## 一、事情是怎么开始的

这次工作的起点，不是直接做项目骨架，而是先解决一个更具体的问题：

**如何让 Emby 通过 `.strm` 正常播放 AList 中的媒体内容。**

当时的上下文里，已经有这些现成条件：

- 现有环境基于 `emby2Alist`
- 已有 AList、Emby、nginx 相关链路
- 已建测试库：`/emby-strm`
- 之前尝试过自定义 `/strm?path=...` 中间入口方案，但结果不稳定

因此，这次并不是“从 0 开始设计一个项目”，而是从一条真实的故障排查线开始，逐步演化成一个独立项目。

---

## 二、最初的关键分歧：到底该怎么写 `.strm`

一开始的争论点主要是：

- `.strm` 里到底应该写什么
- 是继续走自定义 `/strm` 中间入口
- 还是回退到原作者推荐的范式

后来收敛出一个关键方向：

### 原作者推荐的两种 STRM 用法
1. `.strm` 里直接写远程链接
2. `.strm` 里写 **从 AList 根开始的相对路径**

在当前场景中，更值得验证的是：

**AList 根相对路径写法**

当时我们一度把它理解成：

```text
/电影/...
/剧集/...
```

但后面实测发现，这个理解不完整。

---

## 三、第一次关键验证：为什么 `/电影/...` 不行，`/115/电影/...` 才行

实际测试时，先选中了一个简单电影样本：

- `泰坦尼克号`

初始改法是把 `.strm` 内容改成：

```text
/电影/泰坦尼克号/Titanic....mp4
```

结果是：

- Emby 能识别条目
- 媒体信息里也能看到 STRM 内容
- 但是 **播放失败**

这一步非常重要，因为它说明：

### 失败点不在入库，而在播放解析

也就是：
- Emby 已经吃到 `.strm`
- 但当前链路不能把 `/电影/...` 正确解析成可播放资源

接着改成：

```text
/115/电影/泰坦尼克号/Titanic....mp4
```

这次播放成功。

### 由此得到的核心结论
当前环境里的所谓 “AList 根相对路径”，它的“根”不是：

- `/电影`

而是：

- `/115`
- `/ali`
- `/quark`
- 等存储根

也就是说，正确结构应该理解为：

```text
/{storage_root}/{category_root}/...
```

这是这次项目里最关键的一条经验结论之一。

---

## 四、从“电影/剧集”写死，升级到变量化思路

在验证通过后，又很快意识到另一个问题：

不止 `/115` 不能写死，连：

- `/电影`
- `/剧集`

也不能写死。

因为真实环境里将来会出现：

- `/动画`
- `/纪录片`
- `/综艺`
- 甚至完全自定义的分类目录

所以路径模型被进一步抽象成：

```text
/{storage_root}/{category_root}/{relative_media_path}
```

这个抽象使得项目从“只解决当前 115 电影问题”，提升成“可扩展到多盘、多分类”的通用模型。

---

## 五、为什么最后选择了镜像型方案

在路径模型明确后，又出现了一个更大的设计问题：

### 本地 STRM 库目录结构，到底怎么组织？

主要有两种方案：

#### 方案 A：镜像型
AList 是什么结构，本地 STRM 就镜像成什么结构。

例如：

AList 真路径：
```text
/115/电影/泰坦尼克号/Titanic....mp4
```

本地 STRM：
```text
/emby-strm/115/电影/泰坦尼克号/Titanic....strm
```

#### 方案 B：展示型重组
本地按照 movies-115、tv-115 等更“媒体库友好”的方式重组。

最后选择了：

## **镜像型方案**

原因很明确：

1. 最少映射层
2. 最容易排障
3. 路径一一对应
4. 更适合自动化生成
5. 后续支持多盘多分类也最自然

这个决策，是项目从“单点修问题”走向“独立工具设计”的分水岭。

---

## 六、复杂样本为什么必须测

简单样本成功后，并没有直接进入批量生成，而是专门测了复杂样本。

因为真实媒体库里存在大量这种路径：

- 多层级合集目录
- 超长中文目录名
- 空格、括号、全角符号
- 中英文混合
- 一层合集目录下再分片名目录

例如典型路径：

```text
/emby-strm/115/电影/0《漫威宇宙系列35部合集》收藏版【...】/钢铁侠 3 .../Iron.Man.3....strm
```

这个复杂样本最终也验证通过。

### 这一步的意义
它证明镜像型方案不只适用于简单单片，也能覆盖真实库里的复杂目录结构。

这使得“批量生成”从理论上变成现实可行。

---

## 七、从“手工改样本”走向 `xstrm`

当简单样本和复杂样本都打通后，项目自然进入下一阶段：

### 不能再手工一条条建 `.strm`

于是开始做命令行工具，目标是：

- 以后新增影片时，不必手工建 `.strm`
- 能扫描指定目录
- 能从已发现目录里选择扫描
- 能做增量生成
- 后续能做成类似 `x-ui` 的命令行管理模式

于是有了第一版：

## `strm-x`

后来名字又统一改成：

## `xstrm`

命名调整的原因是：

- `xstrm` 更像一个独立工具名
- 更符合以后作为主命令发布
- 更适合和 `xstrm-admin` 配套

---

## 八、为什么后来强行要求“和 emby2alist 同源整合”

最初在做 `xstrm` 时，有一段时间其实是自己在生成器里独立推导路径。

但后来很快发现，这样风险很大：

- `emby2alist` 已经有一套成熟的路径处理逻辑
- 当前播放能通，正是因为这套逻辑在生效
- 如果 `xstrm` 再造一套平行规则，后面一定会分叉

于是项目方向被纠正成：

## `xstrm` 必须尽量复用 `emby2alist` 的配置逻辑

具体体现在：

- 读取 `mediaMountPath`
- 读取 `alistAddr`
- 读取 `alistToken`
- 读取 `alistPublicAddr`
- 用挂载路径推导扫描源路径
- 用去挂载前缀后的路径生成 `.strm`

这一步是从“工具能用”升级到“工具和现有生产链路一致”的关键。

---

## 九、为什么后来又改成“默认 Docker 运行 nginx/emby2alist”

在做独立项目过程中，又遇到一个架构级问题：

### nginx 到底怎么运行？

一开始项目里同时做了：

- 宿主机 nginx 安装检查
- nginx 控制脚本
- nginx 配置渲染

但后来结合现有环境重新思考，发现：

### 默认让用户把 nginx 装到本机，并不是最优解

原因很现实：

- 很多用户的 VPS 已经有现成 nginx
- 有的网站是宝塔、站点、反代等复杂环境
- 直接装宿主机 nginx，容易端口冲突和配置冲突
- 用户本身可能已经习惯 docker 跑 nginx

结合你自己的现状，最终定稿为：

## 主方案
- `xstrm / xstrm-admin`：本机安装
- `nginx + emby2alist`：Docker 运行
- `AList / Emby`：外部依赖接入

这个决策把项目的部署路线正式定了下来。

---

## 十、为什么 `runtime.yaml` 是整个项目的关键

随着项目越来越复杂，单纯改几个脚本已经不够了。

必须要有一个统一配置真源，来承载：

- Emby 地址/API
- AList 地址/Token
- nginx 端口/HTTPS/证书
- 媒体挂载根路径
- xstrm 输出目录与源列表

于是建立了：

## `config/runtime.yaml`

它的目标是成为：

### 唯一主配置源
然后再派生出：
- `config/strm-sync.yaml`
- `nginx/conf.d/constant.js`
- `nginx/conf.d/config/constant-mount.js`
- `docker-compose.yml`
- HTTP / HTTPS 站点配置

这一步是把项目从“会跑的一堆脚本”推进到“真正可维护的配置驱动系统”。

---

## 十一、为什么要做模板渲染 + apply

有了 `runtime.yaml` 以后，不能只是放在那里好看。

于是后面又做了：

### 1. 模板系统
包括：
- `constant.js.template`
- `constant-mount.js.template`
- `strm-sync.yaml.template`
- `site-http.conf.template`
- `site-https.conf.template`
- `docker-compose.yml.template`

### 2. 渲染脚本
- `render_runtime.py`

### 3. 应用脚本
- `apply_runtime.py`

这样形成完整链路：

```text
runtime.yaml
  -> render
  -> 运行配置文件
  -> apply
  -> 实际启用配置
```

这一步使项目从“可编辑”变成“可部署”。

---

## 十二、为什么 HTTP / HTTPS 切换被单独做成能力

又一个很关键的问题是：

### 如果用户一开始装的是 HTTP，后面想切 HTTPS，要不要重装？

最终结论是：

## 不需要重装

因为当前架构已经决定：

- nginx 跑在 Docker 容器里
- 配置来自挂载目录
- runtime 控制 HTTP / HTTPS 模式

所以 HTTP → HTTPS / HTTPS → HTTP，本质上只是：

- 改配置
- 重新渲染
- 重新应用
- 重启容器

而不是重新安装 nginx。

后来这件事被正式写成：

- 文档：`docs/HTTP_HTTPS_SWITCH.md`
- 管理入口：`xstrm-admin` 中的重新渲染与应用

这一步对“可维护性”意义很大。

---

## 十三、为什么 AList 地址后来改成 host/port 分离

在安装交互设计时，又意识到一个真实问题：

### 不能把 AList 地址整段写死成 URL

因为很多人：
- Docker 装 AList
- 本机装 AList
- 自定义端口
- 域名反代

所以后面改成：

## AList 配置分离成
- `scheme`
- `host`
- `port`
- `token`
- `public_addr`

并且安装时先问：
- Docker / 本机 / 自定义

再给不同默认建议值。

这是从“能填”升级到“适合真实部署”的一步。

## 十五、今天这次折腾最终沉淀下来的核心结论

### 结论 1
`.strm` 正确写法是：

```text
/{storage_root}/{category_root}/{relative_media_path}
```

而不是单纯 `/电影/...`。

### 结论 2
本地 STRM 库采用：

## 镜像型方案

即 AList 目录结构镜像到 `/emby-strm`。

### 结论 3
`xstrm` 必须尽量与 `emby2alist` 同源整合，不能自己另造一套路径系统。

### 结论 4
默认部署主方案应当是：
- `xstrm / xstrm-admin` 本机安装
- `nginx + emby2alist` Docker 运行
- `AList / Emby` 外部接入

### 结论 5
从 HTTP 切 HTTPS，不属于重装，而属于配置切换。

### 结论 6
当前项目更适合：
- 先私有仓库继续打磨
- 再决定是否公开发布
