# xstrm-suite 历史设计结论与近期运维经验（2026-03 整理版）

## 文档目的

这份文档用于把两部分内容收敛到一处：

1. 项目最初从真实问题中收敛出来的**历史设计结论**
2. 2026-03 这几天在真实环境里继续打磨得到的**近期运维经验**

目标不是重复参数手册，而是形成一份对后续维护最有价值的“判断依据文档”：

- 为什么当初这么设计
- 现在为什么继续这么维护
- 哪些地方已经被真实验证
- 遇到问题时应该先从哪里判断

---

## 一、项目原点：它不是先有方案，后有问题

`xstrm-suite` 不是从抽象架构设计直接开始的。

它真正的起点是一个非常具体的需求：

> **让 Emby 通过 `.strm` 稳定播放 AList 中的媒体资源，并且后续维护、迁移、批量生成都足够顺手。**

所以，项目最核心的问题从来不是“做一个新工具”，而是：

- `.strm` 到底该怎么写
- AList 根路径到底该怎么理解
- 是否要继续走自定义 `/strm` 中间入口
- 如何兼容多盘、多分类、复杂目录
- 如何把手工验证收敛成可重复部署

这决定了一个很重要的判断原则：

> **后续凡是要调整 xstrm-suite 的结构，应该先回到这个原始问题，而不是为了功能看起来更全而盲目扩张。**

---

## 二、历史上最关键的一次验证：路径语义到底是什么

### 1. 当时的关键现象

历史验证里，出现过这样一组非常关键的对比：

- `.strm` 写成 `/电影/...`
  - Emby 能识别条目
  - 也能看到 STRM 内容
  - **但播放失败**

- `.strm` 写成 `/115/电影/...`
  - **播放成功**

这个结论非常关键，因为它把问题切开了：

> **如果 Emby 能识别条目但播放失败，问题往往不在入库，而在播放解析链路。**

也就是说：

- Emby 已经“吃到了” `.strm`
- 失败点不在扫描
- 而在后续链路没能把 `.strm` 里的路径正确解析成可播放资源

---

### 2. 由此得出的路径模型结论

这次验证最终收敛出的路径模型是：

```text
/{storage_root}/{category_root}/{relative_media_path}
```

例如：

```text
/115/电影/泰坦尼克号/Titanic....mp4
```

这里最容易误解的点是：

> 所谓 “AList 根相对路径” 的根，通常不是 `/电影` 这类业务分类目录，
> 而是 `/115`、`/ali`、`/quark` 这类**存储根**。

这条结论是整个 xstrm-suite 历史设计里最重要的基础之一。

后续如果再遇到：

- 条目能扫出来
- 但播放失败
- 或者 302 / 解析链路异常

应优先回到这条路径语义检查，而不是先怀疑 Emby 本身。

---

## 三、为什么最终采用“镜像型 STRM”

历史设计最终没有选择“只把文件生成出来就行”的思路，而是采用了：

## 镜像型 STRM

即：

- AList 是什么结构
- 本地 STRM 库就尽量镜像成什么结构

这套方案的价值，不在于“好看”，而在于它在真实维护里最稳：

### 镜像型方案的好处

1. **路径关系最直观**
   - 看到本地目录，就能迅速映射回 AList 真路径

2. **排障最容易**
   - 一旦播放异常，可以快速判断是：
     - 存储根错了
     - 分类根错了
     - 相对媒体路径错了
     - 挂载语义错了

3. **迁移成本更低**
   - 不需要维护一套额外的“映射脑图”
   - 项目迁移到别的机器时更容易恢复

4. **批量生成更自然**
   - 多盘、多分类、未来扩展自定义目录时，不需要为每类目录重写规则

因此，这不是一个临时实现细节，而是当前方案的核心设计原则之一。

---

## 四、为什么不再继续走自定义 `/strm` 中间入口

项目早期曾考虑继续走自定义 `/strm?path=...` 之类的中间入口。

但历史验证后，最终并没有把它作为主方案固定下来。

原因不是“不能做”，而是：

- 中间层会再造一套额外解析规则
- 路径语义会变复杂
- 排障时多了一层不必要的不确定性
- 实际可维护性不如直接遵循 `emby2alist` 现有语义

所以项目最终收敛到的方向是：

> **尽量复用 `emby2alist` 已有的路径与播放规则，而不是再造一套平行规则。**

---

## 五、xstrm-suite 的长期架构原则

### 1. 复用 `emby2alist`，而不是与它“各玩各的”

历史设计已经明确：

`xstrm` 不应该和现有播放链路各玩各的。

因此当前成熟方向应保持为：

- 同源读取 `emby2alist` 的关键配置
- 同步挂载路径语义
- 统一 AList 地址和播放路径逻辑
- 尽量避免再发明平行规则

这意味着：

> **xstrm-suite 的价值，不是替代 `emby2alist`，而是把 `.strm` 生成、扫描、部署、配置编排统一起来。**

---

### 2. 项目边界要保持清晰

当前已经明确的边界是：

- `xstrm / xstrm-admin`：本机安装
- `nginx + emby2alist`：Docker 运行
- `AList / Emby`：外部依赖接入

所以，`xstrm-suite` 更准确的定位是：

## 一个独立的“媒体链路管理与部署方案”

它主要负责：

- `.strm` 生成
- `.strm` 扫描与任务化操作
- nginx / emby2alist 配置管理
- Docker 运行编排
- 配置渲染与应用
- 安装、升级、修复、卸载入口

它**不是**：

- AList 替代品
- Emby 替代品
- 全家桶式媒体平台安装器

后续如果项目继续演进，仍应守住这个边界，否则维护成本会快速失控。

---

## 六、为什么默认主方案不是“全部装宿主机”

历史设计中途确实考虑过：

- 直接在宿主机安装 nginx
- 把所有东西都压到同一层管理

但最终定稿没有这么做，而是选择：

- `xstrm / xstrm-admin`：本机安装
- `nginx + emby2alist`：Docker 运行
- `AList / Emby`：外部依赖

原因在真实环境里很现实：

1. 很多机器已经有自己的 nginx
2. VPS 上经常已有现成站点和反代
3. Docker 更方便迁移、隔离和回滚
4. 这种结构更贴近真实现有环境

所以这不是“技术洁癖式分层”，而是现实运维下更稳的默认方案。

---

## 七、为什么 `runtime.yaml` 是关键里程碑

随着项目能力增加，继续靠多个文件手工改配置，会迅速失控。

因此项目历史上一个重要收敛点是：

## 引入统一主配置 `config/runtime.yaml`

由它统一派生：

- xstrm 运行配置
- nginx / emby2alist 配置片段
- Docker compose
- HTTP / HTTPS 站点配置

这意味着项目已经从：

- 脚本拼装阶段

进入到：

- **配置驱动阶段**

这条原则对后续维护非常重要：

> **优先改主配置，再渲染、再应用，不要直接散改派生文件。**

否则很容易出现：

- 当前改好了
- 下次渲染又被覆盖
- 结果形成“看似修好，实际不可持续”的状态

---

## 八、为什么 HTTP / HTTPS 切换应该被视为“配置问题”

历史设计里还有一个重要结论：

> HTTP → HTTPS 的切换，不应该被理解成“重装问题”，而应该被理解成“配置切换问题”。

正确方式应是：

- 改 `runtime.yaml`
- 重新渲染
- 重新应用
- 重启相关容器/服务

而不是：

- 重装 nginx
- 重做整套 Docker
- 手工覆盖一堆历史文件

这条经验很重要，因为它决定了项目是否真的具备长期维护能力。

---

## 九、近期真实运维经验（结合这几天的实际修复）

下面这些不是历史抽象设计，而是最近几天已经在当前环境里踩实的运维经验。

### 1. `profile_root` 默认旧路径问题必须彻底清干净

近期已修复：

- 管理页默认 `profile_root` 仍指向旧路径 `/root/emby2Alist/nginx`
- `scripts/load_nginx_profile.py` 默认 `--root` 仍指向旧路径
- `scripts/admin_api.py` 返回/接收 `profile_root` 仍使用旧默认值

这类问题说明：

> 项目迁移后，最容易出问题的不是主流程，而是**残留默认路径**。

因此每次迁移或重构后，优先检查：

- Web 管理页默认值
- API 默认值
- 脚本参数默认值
- 模板默认路径

---

### 2. `scan_path` HTTP 500 往往不是扫描逻辑本身坏了

近期已验证：

- `scan_path` 在 `/emby-strm` 不存在或不可写时，会导致 HTTP 500

这说明：

> 任务 API 报 500，不一定是业务代码坏了，也可能是部署侧目录不存在、不可写、未挂载。

因此遇到扫描报错时，优先检查：

- 输出目录是否存在
- 目录权限是否正确
- 运行用户是否可写
- 容器是否已正确挂载

---

### 3. Emby 不挂载 `/emby-strm`，就无法直接读取生成结果

近期已验证并修复：

- Emby 容器未挂载 `/emby-strm`
- 导致即使 `.strm` 已生成，Emby 也无法直接读取该目录

这条经验非常重要：

> **生成成功 ≠ Emby 一定能看到。**

还必须确保：

- Emby 视角下也能访问同一条 STRM 目录
- 挂载语义与生成侧保持一致

---

### 4. 当前播放链路已经实测回到 302

近期验证结果：

- `strm_x.py --scan-path` 已可成功生成 `.strm`
- Emby 容器已可读取 `/emby-strm` 中生成的 `.strm`
- 实测播放链路已确认走 302

这说明当前方案不是停留在“生成文件成功”，而是已经回到：

- 生成
- 读取
- 播放解析
- 302 跳转

这条完整链路验证，是近期最重要的实战确认之一。

---

### 5. AList 扫描要保持温和节流，不要打满

近期还补充了扫描侧节流：

- 每个目录请求后随机延时 `1.5~3.5s`
- 每扫描 10 个目录额外停顿 `5~10s`
- 扫描过程增加目录级进度日志

这背后的经验不是“慢一点更优雅”，而是：

> AList 目录扫描属于真实外部链路访问，过猛会带来稳定性和可观测性问题。

因此后续应继续保持：

- 温和节流
- 可见进度
- 任务状态可查询

而不是一味追求扫得最快。

---

## 十、现在遇到问题时，建议优先按这个顺序判断

### 场景 A：条目能出来，但不能播放

优先检查：

1. `.strm` 内容是否仍符合 `/{storage_root}/{category_root}/...`
2. 存储根是否写成了分类根
3. `emby2alist` 配置与路径语义是否一致
4. 当前链路是否还能正确回到 302

---

### 场景 B：扫描接口报错 / 任务失败

优先检查：

1. 输出目录是否存在
2. 输出目录是否可写
3. Emby 是否挂载了 STRM 目录
4. API 默认路径是否仍有旧残留
5. 状态文件 / 日志文件是否可写

---

### 场景 C：迁移或改目录后各种诡异问题

优先检查：

1. `runtime.yaml` 是否仍是唯一主配置来源
2. 派生文件是否被手工改过导致漂移
3. `profile_root` / 默认路径是否仍指向旧位置
4. Docker 挂载语义是否和生成侧一致

---

## 十一、当前阶段的文档化结论

截至这次整理，可以把 `xstrm-suite` 当前状态理解为：

### 已经不是试验脚本，而是一套可持续打磨的私有媒体链路方案

它最重要的设计支柱是：

1. **路径模型清晰**
   - `/{storage_root}/{category_root}/{relative_media_path}`

2. **镜像型 STRM**
   - 本地结构尽量镜像 AList 真路径

3. **复用 emby2alist 既有能力**
   - 不再自造平行规则

4. **配置驱动**
   - 以 `runtime.yaml` 为统一主配置入口

5. **默认部署边界清晰**
   - xstrm 本机、nginx/emby2alist Docker、AList/Emby 外部接入

6. **近期运维链路已重新打通**
   - `.strm` 生成正常
   - Emby 可读取
   - 播放链路回到 302

---

## 十二、后续维护建议

### 继续坚持的方向

- 继续把 `runtime.yaml` 作为唯一主配置入口
- 保持镜像型 STRM 结构，不轻易引入新的路径抽象层
- 尽量复用 `emby2alist` 规则，不新增平行语义
- 新增功能时先问：是否真的属于“链路管理与部署方案”的边界内

### 尽量避免的方向

- 重新引入复杂 `/strm` 中间层作为主方案
- 直接散改派生配置文件
- 把项目膨胀成 AList / Emby 全家桶安装器
- 忽视默认路径残留问题
- 只验证“生成成功”，不验证 Emby 读取与播放跳转

---

## 十三、一句话版本

> **xstrm-suite 的正确维护方式，不是不断加功能，而是守住路径语义、镜像型结构、emby2alist 复用和 runtime 配置驱动这四条主线。**
