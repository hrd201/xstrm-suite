# xstrm-suite 项目记忆维护方案

## 目标

`xstrm-suite` 不是一次性脚本，而是会持续细化的长期项目。

因此，项目记忆不能只依赖聊天上下文，而要形成可维护、可回放、可召回的三层结构。

---

## 一、三层记忆结构

### 1. 过程层（Daily Memory）
记录当天发生了什么。

写入位置：
- `memory/YYYY-MM-DD.md`

适合记录：
- 今天做了哪些改动
- 遇到了什么问题
- 为什么临时改成现在这样
- 哪些问题还没收口
- 当前下一步是什么

特点：
- 时间线清晰
- 允许有过程性内容
- 不要求完全提炼

---

### 2. 长期结论层（Long-term Memory）
记录稳定、可复用、跨天仍然重要的结论。

写入位置：
- `MEMORY.md`
- 向量记忆（`memory_store`）

适合记录：
- 已确认的路径语义
- 已固定的部署模型
- 已确定的产品命名和边界
- 多次验证过的规则
- 经常重复出现的坑点

特点：
- 必须短、小、稳
- 不写流水账
- 优先原子化表达

---

### 3. 项目文档层（Repo Docs）
记录项目本身应该长期保留的说明。

写入位置：
- `projects/xstrm-suite/docs/`
- `README.md`

适合记录：
- 设计思路
- 部署方式
- 演化过程
- 发布说明
- 结构说明

特点：
- 面向未来维护
- 面向仓库阅读者
- 不依赖聊天上下文

---

## 二、xstrm 项目优先沉淀的内容

后续只要出现以下内容，就应优先沉淀：

### A. 设计决策
例如：
- 为什么采用镜像型 STRM
- 为什么命令名定为 `xstrm`
- 为什么 `runtime.yaml` 是统一真源
- 为什么默认运行方式是“本机 xstrm + Docker nginx/emby2alist + 外部 AList/Emby”

### B. 路径与播放规则
例如：
- `.strm` 内容必须遵循什么路径语义
- 哪些路径在当前环境能入库但不能播放
- 哪些字段必须与 `emby2alist` 保持同源

### C. 部署与运维规则
例如：
- HTTP/HTTPS 切换规则
- Docker 与宿主机的边界
- bootstrap / render / apply 的职责分工

### D. 重复坑点
例如：
- 不要重新发明一套和 `emby2alist` 不一致的路径规则
- 不要把测试库误当生产模板
- 不要把运行时生成物和源模板混淆

---

## 三、执行规则

### 每次重要变更后至少做两件事
1. 写入当天 `memory/YYYY-MM-DD.md`
2. 判断是否需要补一条长期结论记忆

### 当出现以下情况时，必须优先 recall
- 重复报错
- 路径语义重新讨论
- 部署模型重新调整
- 和 `emby2alist` 的关系再次发生变化
- 遇到“之前明明讨论过”的问题

### 文档优先级
如果一个结论会影响：
- 后续代码改动
- 安装方式
- README 表达
- 目录结构

那么它不应只留在聊天里，必须沉淀到文档或长期记忆。

---

## 四、当前建议的后续动作

1. 持续把 `xstrm` 的新决策写入 daily memory
2. 逐步把稳定结论提炼进长期记忆
3. 后续再经用户确认后，补充 `MEMORY.md` 中的 xstrm 专项长期记忆块
4. 继续完善仓库 `docs/`，让项目知识尽量脱离聊天上下文也能完整理解

---

## 一句话原则

`xstrm-suite` 的记忆策略应该是：

**过程写 daily，规则写长期，说明写文档。**
