# 项目由来与设计思路

## 项目由来

`xstrm-suite` 不是从抽象设计直接开始的，而是从一个真实问题逐步长出来的：

**如何让 Emby 通过 `.strm` 稳定播放 AList 中的媒体资源，并且后续维护、迁移、批量生成都足够顺手。**

项目最初的关注点并不是“做一个新工具”，而是：

- `.strm` 到底该怎么写
- AList 根路径到底怎么理解
- 是否要继续走自定义 `/strm` 中间入口
- 如何兼容复杂目录、多盘、多分类
- 如何从手工验证走向可重复部署

随着问题不断被拆开和解决，最后逐步沉淀成了一个完整方案。

---

## 设计目标

### 1. 让 `.strm` 写法尽量简单、稳定、可解释
最终采用：

```text
/{storage_root}/{category_root}/{relative_media_path}
```

例如：

```text
/115/电影/泰坦尼克号/Titanic....mp4
```

---

### 2. 让本地 STRM 库结构与 AList 真路径尽量一致
最终采用：

## 镜像型方案

即：
- AList 是什么结构
- 本地 STRM 就镜像成什么结构

这样路径关系最清楚，排障最容易。

---

### 3. 尽量复用 `emby2alist` 现有能力，而不是再造一套平行规则
`xstrm` 不应该和现有播放链路各玩各的。

所以项目后面明确走向：

- 同源读取 `emby2alist` 的关键配置
- 同步挂载路径语义
- 统一 AList 地址和播放路径逻辑

---

### 4. 兼顾维护与迁移
这决定了项目最终必须具备：

- 统一主配置
- 模板渲染
- 安装入口
- 管理入口
- Docker 运行方式
- HTTP / HTTPS 可切换

---

## 为什么不是“全部装宿主机”

项目中途曾经考虑过直接安装宿主机 nginx。

但结合真实使用环境后，最终定稿为：

## 默认主方案
- `xstrm / xstrm-admin`：本机安装
- `nginx + emby2alist`：Docker 运行
- `AList / Emby`：外部依赖接入

原因很简单：

- 很多机器已经有自己的 nginx
- VPS 上经常已有站点和反代
- Docker 更方便迁移、隔离和回滚
- 更贴近真实现有环境

---

## 为什么要有 `runtime.yaml`

随着能力增加，配置不能再分散在很多文件里手工改。

所以建立了统一主配置：

- `config/runtime.yaml`

由它统一派生：

- xstrm 运行配置
- nginx / emby2alist 配置片段
- Docker compose
- HTTP / HTTPS 站点配置

这一步使项目进入“配置驱动”的阶段。

---

## 为什么 HTTP / HTTPS 切换是配置问题，而不是安装问题

项目后期明确了一个重要原则：

### HTTP -> HTTPS
不需要重装 nginx，也不需要重装 Docker。

只需要：
- 改 `runtime.yaml`
- 重新渲染
- 重新应用
- 重启容器

这使得项目更符合长期维护的预期。

---

## 当前项目定位

`xstrm-suite` 当前更适合被理解为：

### 一个独立的“媒体链路管理与部署方案”
而不只是一个生成 `.strm` 的小脚本。

它解决的是整条链路：

- `.strm` 生成
- AList / Emby 接入
- nginx / emby2alist 运行配置
- Docker 运行方式
- 配置渲染与应用
- 安装、升级、修复、卸载
